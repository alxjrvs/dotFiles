## Agent secret access

The agent resolves 1Password secrets through a **service account**, not your desktop biometric session — so a Claude session (interactive *or* headless/cron) gets its secrets with **no Touch ID prompt** and **no dependency on the 1Password desktop app**. This is the 1Password-recommended automation tier ([secure-ai-access](https://www.1password.dev/get-started/secure-ai-access), [developer-quickstart](https://www.1password.dev/get-started/developer-quickstart)).

- **Scope = one vault.** The service account can read only the dedicated `claude-agent` vault (1Password forbids granting a service account access to Personal/Private, which is *why* agent secrets live in their own vault). That vault is the entire blast radius.
- **Token lives in the macOS login keychain** (`security … -s op-claude-agent`), never on disk in plaintext, never in git. `botu apply` runs `op-agent provision`, which creates the vault + service account and stores the token on first run.
- **The token never enters the model's context.** It's read from keychain *inline* inside the `op-agent` CLI (`op-agent header op://…`, the MCP `headersHelper`), confined to that one `op` process — so neither the service-account token nor the resolved secret reaches a Bash subprocess, the transcript, or OTEL tool spans. No `claude()` wrapper, no exported env var. (`op-agent` is one verb-dispatched script — `header`/`provision`/`status` — replacing the old per-service shims.)
- **MCP secrets follow one canonical pattern.** For servers we install, the launch command is 1Password's `op run --env-file=.env -- <server>` (`botu mcp add`) with `op://` references in a committable `.env` — resolved in-process, off disk. Two surfaces we don't control use the tool's *own* native hook fed by `op`: HTTP servers via Claude Code's `headersHelper` (above), plugin-bundled stdio servers via a `*_COMMAND` resolver var (e.g. spacebase). A resolved secret never enters git, and never write a `${VAR}` into a git-tracked `.mcp.json` (`claude mcp add` can expand it back into the tracked file). Full model in the dotFiles repo `CLAUDE.md`.
- **Agent git auth uses a cached PAT, not `op`.** The agent's git-over-HTTPS credential is a least-privilege fine-grained PAT that `botu apply` (`op-agent provision`) caches in the login keychain; a Claude session reads it via the stock `osxkeychain` helper (no biometric, headless-safe), keeping the agent off your broad `gh` OAuth token.
- **Your own dev work** still uses desktop biometric + `op run`/`op://`/Environments — the service account is the agent's path, not yours. Full secrets model in the dotFiles repo `CLAUDE.md`.
